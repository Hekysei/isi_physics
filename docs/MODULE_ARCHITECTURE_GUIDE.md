# Руководство по модульной архитектуре для переносимых проектов

## Цель архитектуры

Сделать модуль самодостаточным внутри собственного пространства имен (feature scope), чтобы:

- код не конфликтовал с другими модулями;
- перенос выполнялся копированием директорий `src/main` и `src/test`;
- изолированные проекты как пазлы встают в общую картину и не затрагивают другие части;


## Базовые правила

- У каждого модуля есть уникальный идентификатор (например, `featureX`, `kirchhoff`'), который используется в:
  - путях java файлов,
  - путях шаблонов,
  - путях статических ресурсов,
  - путях тестов.
- Все, что относится к модулю, хранится только внутри его папки.
- Весь браузерный роутинг (контроллеры, редиректы, ссылки, формы) должен быть строго под префиксом `/featureX/...`; модуль не должен маппиться на `/` и любые другие корневые пути, не относящиеся к его namespace.

## Рекомендуемая иерархия каталогов

```text
project-root/
└── src/
    ├── main/
    │   ├── java/
    │   │   └── com/example/isib/
    │   │       └── featureX/*
    │   └── resources/
    │       ├── templates/
    │       │   └── featureX/*
    │       ├── static/
    │       │   └── featureX/*
    └── test/
        ├── java/
            └── com/example/isib/
                └── featureX/*
```

## Что нельзя делать
- Хранить классы/шаблоны/статику/тесты в общих папках без `featureX` поддиректории.
- Делать маппинг браузерных адресов на `/` или перехватывать чужие пути вида `/<что-угодно>` вне `featureX`.


## Мини-чеклист перед переносом

- [ ] Весь код модуля находится в `src/main/.../featureX`.
- [ ] Все тесты модуля находятся в `src/test/.../featureX`.
- [ ] Все ресурсы лежат в `src/resources/( templates , static )/featureX`.
- [ ] Весь маппинг адрессов строго через `/featureX`
